# Build Spec — 2FA "Trouble with Authenticator" Flow + Notifications Panel

Source of truth for the mobile PWA prototype rebuild. Derived from web-dashboard screenshots in:

- `reference/screenshots/Trouble with Authenticator/` (Steps 1–4)
- `reference/screenshots/Notification/` (Step 1)

Design system: components in `src/components/`, tokens in `src/styles/tokens.css`. The builder must NOT consult the screenshots — everything needed is in this document.

## Flow A — Trouble with Authenticator (2FA fallback via Email + SMS OTP)

All four web screens share the same shell: a centered white card (radius ~12px, `--radius-lg`, shadow `--shadow-2/3`) floating over a full-bleed dark-teal photographic/brand background (deep ivy `#002b2a`-toned cityscape with vibrant-green `#00d37e` logo swashes). Every card starts with the **StraitsX logo** (green infinity logomark + "StraitsX" wordmark, `Logo` component) centered at the top. Web URLs live under `sso.straitsx.com/auth/realms/xfers-sg/...` — this is the Keycloak SSO surface, i.e. an unauthenticated context (no nav, no sidebar).

Mobile PWA adaptation (applies to all of Flow A): render each step as a **full-screen auth page** (not a modal — there is no underlying page to reveal on mobile SSO), stacked vertically, content max-width ~400px centered, `--background` or brand-dark backdrop with a white `Card`-like sheet. Alternatively reuse `Modal2FA`'s inner layout parts as a page. Details per screen below.

### Screen A1 — "Enter Verification Code" (authenticator TOTP entry)

**Purpose.** The standard 2FA gate during login: user enters the 6-digit TOTP from their authenticator app. This screen is the *entry point* to the trouble flow via its bottom link.

**Layout (web).** Single centered card, vertical stack, all content center-aligned:
1. StraitsX logo (green logomark + wordmark).
2. Heading (H1, dark ivy, bold, ~28px): `Enter Verification Code`
3. Body copy (grey-dark, ~16px, center): `Enter the 6-digit code generated by your Two-factor Authentication (2FA) app.` — the phrase **"Two-factor Authentication (2FA) app"** is bold; rest is regular weight.
4. OTP input: **6 single-digit fields**, underline style (no boxes — a light-grey bottom border per digit, ~48px wide each, ~16px gap; the focused/filled digit's underline turns green `--primary`; text-caret visible in the active cell; digits rendered large ~28px, dark ivy, centered).
5. Primary button, full card-content width: label `Submit` — solid vibrant green `--primary`, dark-ivy text, pill/8px radius, ~48px tall (`Button variant="primary" size="lg"`). In the screenshot the code is complete, so Submit is enabled.
6. Text link below the button, centered, credible-blue `--link`: `Having trouble with authenticator?` (no icon, no underline at rest).

**Every element / verbatim copy.**
| Element | Copy (verbatim) | Component mapping |
|---|---|---|
| Logo | — | `Logo` |
| Heading | `Enter Verification Code` | `.title-large` / Modal2FA title |
| Instruction | `Enter the 6-digit code generated by your Two-factor Authentication (2FA) app.` | Modal2FA `instruction` (matches its default) |
| OTP field ×6 | pre-filled demo digits `4 1 6 3 7 0` | `Modal2FA` segmented code input (`length={6}`) |
| Button | `Submit` | `Button variant="primary" size="lg"` (note: label is **Submit**, not Modal2FA's default "Verify" → pass `verifyLabel="Submit"`) |
| Link | `Having trouble with authenticator?` | Modal2FA `troubleText` (matches default) / `LinkButton` |

**Dummy data.** Entered code: `416370`. URL (context only): `sso.straitsx.com/auth/realms/xfers-sg/login-actions/authenticate?execution=3e64c250-7e85-4785-a829-184de69f30c7&client_id=xfers-app&tab_id=iZS…`

**Flow logic.**
- Tap `Submit` with valid code → normal login completes (out of scope here).
- Tap `Having trouble with authenticator?` → **Screen A2 (Security check)**.
- Submit stays disabled until all 6 digits are entered (mirror `Modal2FA`'s `value.length < length` rule).

**Mobile adaptation.** Full-screen page. Use `Modal2FA` with `verifyLabel="Submit"`, `onTrouble` wired to navigate to A2 — or lift Modal2FA's hero/instruction/code/trouble sub-layout into a page shell. Single full-width Submit (no Cancel on this first screen). Auto-advance focus between digits; `inputMode="numeric"`; support paste of full code (Modal2FA already does).

### Screen A2 — "Security check"

**Purpose.** Explains the fallback: StraitsX will send OTP codes to the user's verified channels (email + phone) to re-verify identity before resetting/bypassing the authenticator. This is a confirmation/consent step — nothing to fill in.

**Layout (web).** Same centered card:
1. StraitsX logo.
2. Heading: `Security check`
3. Body copy (center): `We will send you OTP codes for the following verifications to proceed.`
4. A two-row **checklist**, left-aligned within the card, each row separated by full-width 1px light-grey hairline dividers (above row 1, between rows, below row 2):
   - Row 1: green check icon (✓, `--primary`/`--status-positive`) + label `Email` (grey-dark body text, ~16px)
   - Row 2: green check icon + label `Phone number`
   - Rows are ~80px tall, icon and label vertically centered, icon leading with ~16px gap.
5. Footer button row, two buttons side by side (~48% width each, gap between):
   - `Cancel` — white/outlined: white surface, 1px light-grey border, teal text `--btn-secondary-text` (`Button variant="secondary" size="lg"`)
   - `Continue` — solid green primary (`Button variant="primary" size="lg"`)

**Verbatim copy.** Heading `Security check`; body `We will send you OTP codes for the following verifications to proceed.`; list items `Email`, `Phone number`; buttons `Cancel`, `Continue`.

**Dummy data.** None beyond the copy. URL context: `.../login-actions/reset-credentials?client_id=xfers-app&tab_id=L3LznaMe5nc&reset_cred_type=totp`.

**Flow logic.**
- `Continue` → sends email OTP → **Screen A3**.
- `Cancel` → returns to **Screen A1** (abandons trouble flow).

**Mobile adaptation.** Full-screen page (keeps flow linear). Checklist rows: use `CardChecklist` if its API fits (check-icon + label rows), otherwise simple list rows with `Icon name="check"` colored `--status-positive` and hairline dividers (`border-bottom: 1px solid var(--border)` at reduced opacity). Footer: two `Button size="lg"` side-by-side, or stacked full-width on narrow screens (primary on top). An acceptable alternative presentation is a `BottomSheet` pushed over A1, but default to full-page for consistency with the rest of the SSO flow.

### Screen A3 — "Enter Verification Code" (Email OTP)

**Purpose.** User enters the 6-digit OTP sent to their email address. First of the two verifications promised in A2.

**Layout (web).** Same centered card:
1. StraitsX logo.
2. Heading: `Enter Verification Code`
3. Body copy: `Enter the code that we sent to tewi@fazzfinancial.com` — the email **tewi@fazzfinancial.com** is bold; rest regular.
4. OTP input: 6 underline digit fields, all **empty**; first field focused with visible text caret (green-tinted underline on the focused cell, grey `#d8d8d8`-ish underlines on the rest).
5. Resend line (grey `--text-secondary`, ~14px, center): `Didn't receive the code? Resend in 00:57` — a live **60-second countdown** (mm:ss). After it hits 00:00 it becomes a tappable `Resend` link (blue `--link`) — implement this behavior even though the screenshot only shows the counting state.
6. Footer button row (two buttons, like A2):
   - `Cancel` — secondary/outlined.
   - `Submit` — primary, shown **disabled** (washed-out light green ≈ `#66e5ac`, muted text) because no digits are entered. Enables when 6 digits present.

**Verbatim copy.** Heading `Enter Verification Code`; body `Enter the code that we sent to tewi@fazzfinancial.com`; resend `Didn't receive the code? Resend in 00:57`; buttons `Cancel`, `Submit`.

**Dummy data.** Email: `tewi@fazzfinancial.com`. Countdown start: `00:57` shown (use 60s timer). URL context: `.../reset-credentials?execution=ec09b047-e44f-4589-87ef-5f438060f632&client_id=xfers-app&tab_id=…`.

**Flow logic.**
- Enter 6 digits → `Submit` enables → tap → email verified → **Screen A4** (phone OTP).
- `Cancel` → back to **Screen A1**.
- `Resend` (after countdown) → re-sends email code, restarts timer.

**Mobile adaptation.** Full-screen page reusing the same OTP layout as A1 (`Modal2FA` internals with `instruction` overridden to the "sent to …" copy and `troubleText={null}`). Add the resend-countdown line as a custom element under the code input (grey `--body-medium`, link part in `LinkButton size="md"` once active). Footer uses two buttons (`Cancel` secondary + `Submit` primary, disabled until complete). Consider `autoComplete="one-time-code"` for SMS/email OTP autofill.

**Flow logic.**

## Flow B — Notifications Panel (dashboard)

### Screen B1 — Notifications dropdown over Personal Dashboard Home

**Purpose.** User taps the bell icon in the dashboard top-right; a **notifications dropdown panel** opens anchored below/left of the bell, overlaying the Home page. It lists announcement-type notifications with category filters.

**Underlying page (context — Personal Account Home, web `app.straitsx.com`).** Only needed as backdrop; abbreviated inventory:
- **Left sidebar** (`Sidebar`): StraitsX logo with subtitle `Personal Account`; menu items with leading icons: `Home` (active — light-green pill background `--sidebar-active` #d7efe2, green text/icon `--sidebar-menuitem-text` #257c58, home icon), `Mint`, `Transaction History`, `Support` (headset icon).
- **Top-right cluster**: bell `IconButton` (shown in its **active/pressed state** — light mint-green rounded-square highlight behind it since the panel is open) and profile avatar (grey person-circle) + username `TEWIBOWO`.
- **Main content**: `Estimated Balance ⓘ` label with value `30.59 SGD` (`EstimatedBalance`); `My Assets ↻` card (`CardAsset`/`ListAsset` rows): `XSGD 0.00 ≈ 0.00 SGD`, `XUSD 0.00 ≈ 0.00 SGD`, `USDT 23.77 ≈ 30.59 SGD`, `USDC 0.00 ≈ 0.00 SGD`, each with token mark (`AssetMark`) and partner-logo clusters (some with tiny `NEW` red badge, XSGD row shows `+5` overflow); a `Start your journey with StraitsX` card with segmented buttons `Buy Stablecoin` / `Sell Stablecoin`; right-hand swap panel fragments (`Balance: 0.00 XSGD`, XSGD select, swap-arrows icon button, `Balance: 0.00 XUSD`, XUSD select, `…minute.` rate text); dark-green `StraitsX OTC Desk` promo card (`OtcBanner`): `We offer deep liquidity to institutions and high net-worth individuals.` + `Starting from 100,000 USD…` (partially cut); floating Intercom chat bubble bottom-right; a `History` icon button on the far right edge.
- For the mobile prototype the backdrop can be the already-built Home screen; do not rebuild desktop-only fragments.

**The Notifications panel itself (primary subject).**

Anatomy (web: white dropdown card, ~450px wide, radius ~12px `--radius-lg`, elevation `--shadow-3`, anchored top-right under the bell; inner vertical scroll with visible scrollbar):

1. **Header row**: title `Notifications` (bold, dark ivy, ~20px, `--title-small`) left; close `×` icon button right (`IconButton icon="close" variant="ghost"`).
2. **Filter chip row** (horizontally scrollable; chips truncated at panel edge):

   - `Important` — unselected: light-grey fill `#f0f0f0`-ish, grey-dark text, radius ~6–8px
   - `System Maintenance` — unselected
   - `Product Update` — unselected
   - `OTC Serv…` — unselected, clipped by panel edge; assume full label `OTC Service` (horizontally scrollable strip)
3. **Notification list** — full-width rows separated by 1px hairline dividers; each row:
   - **Leading icon** (~28px, top-aligned): category glyph — System Maintenance = black filled circle with white wrench; Important = amber/yellow warning triangle (`--status-warning`); Product Update = black paper-plane/send glyph.

   - **Body**: regular grey-dark ~15px multi-line (`--body-medium`/`large`), URLs rendered as plain text (not styled links) in the screenshot.
   - No unread dots/highlights are visible — all items read; no timestamps beyond the date; no action buttons on rows.

**Flow logic.**
- Open: tap the **bell icon** in the top nav → panel drops down anchored to the bell (web). Bell shows an active (mint-highlight) state while open.
- Close: tap `×`, tap outside the panel (scrim/blur), or press Escape.
- Filter chips switch the visible list by category; `All` is default. Chips are single-select.
- Tapping a notification row: no visible detail view in the capture — treat rows as non-navigating (static) in the prototype, or expand/collapse if desired (not required).
- Empty filter result → show `EmptyState` (e.g. title `No notifications`, sub `You're all caught up.`) — not in the capture but required for filters with no items.

**Mobile adaptation.**
- Trigger: bell `IconButton icon="notifications"` in the mobile `AppTopNav` (`platform="mobile"`), with unread-count `Badge` via `Badge.Wrap` (e.g. `<Badge tone="critical">3</Badge>` — count arbitrary; screenshot shows no badge, so default 0/none is also acceptable).
- Presentation: **`BottomSheet`** (title `Notifications`, built-in close ×, drag handle) is the recommended mobile pattern; a full-screen page is the fallback if the list is long. Do NOT use a hover dropdown on mobile.
- Filter chips: horizontal scroll row of `Tag` components (`clickable`, `size="small"`, selected chip = green outlined/selected state) directly under the sheet header; `overflow-x: auto`, no wrap.
- List rows: custom `NotificationItem` composed from `Icon` (or `StatusIcon`) + text stack; dividers `1px solid var(--border)` at 50% opacity; row padding `--space-4`/`--space-5`; icon-to-text gap `--space-3`.
- Category icon mapping (Material Symbols via `Icon`): System Maintenance → `build` or `construction` (render white glyph on black `--text-primary` filled circle); Important → `warning` filled, color `--status-warning`; Product Update → `send` filled, color `--text-primary`.
- `Important` meta label color: `--status-critical`; other labels `--text-secondary`.
- Body URLs: plain text (match web), `word-break: break-all` so long ticket URLs wrap.
- Sheet body scrolls internally; keep header + chips pinned.

## Cross-cutting notes

**Component checklist (all exist in `src/components/`):** `Modal2FA` (segmented OTP input, trouble link, Cancel/Verify footer — override `verifyLabel="Submit"`), `Modal`, `BottomSheet`, `Button` (variants primary/secondary, size lg), `LinkButton`, `IconButton`, `Icon`, `Badge` (+ `Badge.Wrap`), `Tag` (clickable/selected for filter chips), `Tabs` (alternative for filters; chips preferred), `Toast`/`ToastProvider` (success feedback), `Alert`, `EmptyState` (title+sub), `AppTopNav` (mobile top bar w/ bell + notifications count), `Logo`/`Logomark`, `Sidebar`, `EstimatedBalance`, `CardAsset`/`ListAsset`, `OtcBanner`, `StatusIcon`, `CardChecklist` (candidate for A2 checklist).

**Tokens / visual system (from `src/styles/tokens.css`):**
- Primary action green `--primary` (#00d37e); pressed/hover `--btn-primary-pressed/hovered`; disabled Submit ≈ washed light-green (use `Button` disabled state).
- Headings: Red Hat Display Bold (`--title-large` desktop / `--title-mobile-large` mobile); body: Hanken Grotesk (`--body-large/medium`).
- Text: `--text-primary` #002b2a; secondary/grey `--text-secondary` #505454; links `--link` #187d97 (screens show link-blue for "Having trouble…", "Contact customer support", filter-none).
- Status colors: warning triangle `--status-warning` #fc9a07; Important label `--status-critical` #df1312; checkmarks `--status-positive`/`--primary`.
- Surfaces: cards/panels `--surface` #fff on `--background` #f6f7f9 (dashboard) or brand-dark imagery (SSO); dividers `--border` #d8d8d8 (hairlines look ~50% opacity); panel radius `--radius-lg` 12px; chips `--radius-md`-ish; buttons pill `--radius-full`.
- Elevation: notification dropdown `--shadow-3`; SSO card `--shadow-2`.
- OTP digits: underline-only cells (transparent field, 2px bottom border; grey idle → green focused/filled), large centered digit, ~48px cell width, `--space-4` gaps.
- Spacing rhythm inside cards: `--space-6` (24) between logo/heading/copy/inputs/buttons; `--space-4` (16) fine gaps.

**Known content quirks to preserve verbatim:** the typo `Syystem Maintenance` (item 1 title); masked phone `*****050`; countdown values `00:57` / `00:54` (both are a 60s timer captured mid-count); email `tewi@fazzfinancial.com`; demo TOTP `416370`.
